# Internationalization (i18n)

Plugins can provide translations for their display text (name, description, parameter labels, etc.). You can declare them directly in `manifest.json` (recommended) or keep using the legacy `locales/*.json` directory.

## Preferred: Inline locales in `manifest.json`

Use i18n keys in fields like `name`, `description`, `category`, `label`, and `group`, then declare the locale dictionaries in `manifest.json`:

```json title="manifest.json"
{
  "name": "meta.name",
  "description": "meta.description",
  "category": "meta.category",
  "params": [
    {
      "key": "speed",
      "label": "params.speed",
      "group": "params.groups.animation"
    }
  ],
  "locales": {
    "en-US": {
      "meta": {
        "name": "Rainbow",
        "description": "Flowing rainbow animation across all LEDs",
        "category": "Animation"
      },
      "params": {
        "speed": "Speed",
        "groups": {
          "animation": "Animation"
        }
      }
    },
    "zh-CN": {
      "meta": {
        "name": "еҪ©иҷ№",
        "description": "жөҒеҠЁзҡ„еҪ©иҷ№еҠЁз”»ж•Ҳжһң",
        "category": "еҠЁз”»"
      },
      "params": {
        "speed": "йҖҹеәҰ",
        "groups": {
          "animation": "еҠЁз”»"
        }
      }
    }
  }
}
```

:::tip
`locales` is the preferred field name. `i18n` and `translations` are also accepted as compatibility aliases.
:::

## Legacy locale files

If you prefer, or need backward-compatible packaging, you can still place translations in the plugin's `locales/` directory:

```
plugins/effect.my_effect/
в”ңв”Җв”Җ manifest.json
в”ңв”Җв”Җ main.lua
в””в”Җв”Җ locales/
    в”ңв”Җв”Җ en-US.json
    в””в”Җв”Җ zh-CN.json
```

## Using i18n Keys

Instead of literal strings in `manifest.json`, use keys that reference entries in your locale dictionaries:

```json title="manifest.json"
{
  "name": "meta.name",
  "description": "meta.description",
  "category": "meta.category",
  "params": [
    {
      "key": "speed",
      "label": "params.speed",
      "group": "params.groups.animation"
    }
  ]
}
```

## Locale Files

Keys like `meta.name` are resolved against either flat dictionaries or nested objects. The examples below use nested JSON because it is easier to read and matches inline manifest locales:

Create JSON files in the `locales/` directory, named by locale code:

```json title="locales/en-US.json"
{
  "meta": {
    "name": "Rainbow",
    "description": "Flowing rainbow animation across all LEDs",
    "category": "Animation"
  },
  "params": {
    "speed": "Speed",
    "preset": "Preset",
    "colors": "Colors",
    "groups": {
      "animation": "Animation"
    }
  }
}
```

## How Resolution Works

1. Core loads legacy `locales/*.json` files first.
2. Core then merges inline manifest locales from `locales` / `i18n` / `translations` on top.
3. If the same locale key exists in both places, the manifest value wins.
4. Core reads the manifest field value (for example `"meta.name"`) and resolves it from the merged locale dictionary.
5. If no match is found, the raw string is used as-is.

This means you can mix literal strings and i18n keys:

```json
{
  "name": "meta.name",
  "publisher": "Skydimo"
}
```

Here `name` uses i18n lookup while `publisher` stays as-is.

## LocalizedText Format (API)

When Core sends localized text over the WebSocket API, it uses the `LocalizedText` format:

```json
{
  "raw": "Rainbow",
  "byLocale": {
    "en-US": "Rainbow",
    "zh-CN": "еҪ©иҷ№"
  }
}
```

The frontend resolves this using `resolveLocalizedText()` based on the user's current language setting.
